# fcr_ros2_3 Project Context

## Default Project Goal

在 `D:\code\fcr_ros2\fcr_ros2_3` 工作区内，所有后续开发、评审、调试、架构设计和文档整理，都默认围绕以下项目目标展开：

本项目旨在基于 ROS 2 Humble 构建一套面向移动跟拍与 Vlog 运镜场景的智能机器人系统。系统以双轮足底盘、DJI RS2 云台、Sony 相机、深度相机、IMU 等硬件为基础，完成从硬件驱动、图像感知、目标跟踪、状态估计、视觉伺服、底盘-云台协同控制，到运镜任务规划和人机交互的完整闭环，使机器人能够稳定识别、锁定并跟随人物目标，同时完成具有摄影表现力的自动构图和运动镜头。

核心技术链：

```text
Sony 图像输入
  -> YOLO 目标检测
  -> 多目标跟踪
  -> 目标状态估计
  -> IBVS / PBVS 视觉伺服
  -> 底盘-云台控制分配
  -> MPC / QP 协同优化
  -> 双轮足底盘 + DJI RS2 云台协同运动
  -> 自动跟随与 Vlog 运镜 Action
```

## Working Assumptions

- 默认 ROS 2 发行版：Humble。
- 代码设计应优先服务完整闭环，而不是孤立 demo。
- 模块边界应围绕硬件抽象、视觉感知、目标状态、伺服控制、任务 Action、人机交互、安全与仿真测试来组织。
- 后续新增功能应尽量兼顾真实硬件和仿真模式切换。
- 控制链路中的安全约束、超时检测、手动接管、速度/角度限幅和可观测状态反馈应作为基础能力处理。

## Development Constraints

后续在本工作区内进行任何代码开发、重构、调试、文档更新或架构决策时，项目目标本身也作为开发约束处理，而不只是背景信息。

具体约束：

1. 每项改动都应能解释它如何服务“智能移动摄影机器人完整闭环”，包括硬件驱动、图像感知、目标跟踪、状态估计、视觉伺服、底盘-云台协同、运镜 Action、人机交互、安全仿真测试中的至少一环。
2. 不为了局部 demo 牺牲主链路一致性；新增接口、话题、参数和节点应尽量接入或兼容主技术链。
3. 优先保持真实硬件与仿真/mock 路径可切换，避免把控制器、感知节点或启动文件写死到单一环境。
4. 涉及运动输出的改动必须考虑安全约束，包括目标丢失、数据超时、急停、限幅、手动接管和零速度回退。
5. 开发顺序应服务当前 V2/V2.5 阶段：先用 mock 数据验证目标状态到控制输出的闭环，再逐步接入真实 YOLO、相机和硬件。
6. 如果某个实现方向偏离最终目标，后续开发前需要主动提醒用户并说明取舍。

## Final Outcome

最终交付目标是一套可运行、可演示、可扩展的 ROS 2 智能移动摄影机器人系统，能够完成：

```text
启动系统
  -> 读取相机图像
  -> 检测并选择人物目标
  -> 持续跟踪目标
  -> 估计目标相对位置和运动状态
  -> 云台保持目标构图
  -> 底盘保持合适拍摄距离
  -> 在安全约束下执行自动跟随
  -> 根据 Action 指令完成推近、拉远、环绕等运镜动作
  -> 通过遥控/语音/Web/Foxglove 进行模式切换和人工接管
```

一句话概括：本项目最终目标是实现一套基于 ROS 2 的智能移动摄影机器人系统，使机器人不仅能稳定识别和跟随人物目标，还能通过底盘、云台、视觉感知和优化控制的协同，完成具有 Vlog 拍摄效果的自动构图与自动运镜。

## V2 Development Flow Reminder

来源：`D:\githubRepository\notes\v2当前阶段开发流程.md`。

后续进入开发前，需要提醒用户：当前 V2/V2.5 阶段优先按 mock-first 流程推进，不急着直接接真实 YOLO 或真实硬件。核心思路是先用 mock 目标数据替代“相机 + YOLO + Tracker + 深度估计”，把“目标状态输入 -> 跟随控制器 -> 底盘/云台控制输出”的链路验证扎实。

结合当前代码，仓库中已经有这条轻量验证链路：

```text
simulation_pkg/scripts/mock_target_publisher_node.py
  -> /target/current (vision_servo_msgs/TargetArray)
  -> servo_control_pkg mvp_follow_controller_node
  -> /cmd_vel + /cmd_gimbal
```

对应启动入口：

```text
ros2 launch servo_control_pkg mvp_mock_test.launch.py scenario:=left
```

当前代码实际使用的是 `/target/current` 和 `vision_servo_msgs/msg/TargetArray`，不是流程笔记中泛化示例里的 `/target/state` 或 `fcr_msgs/msg/TargetState`。后续提醒或开发时应以当前仓库接口为准。

优先验证顺序：

1. 只模拟水平像素偏差，验证云台 yaw 控制方向。
2. 只模拟垂直像素偏差，验证云台 pitch 控制方向。
3. 模拟 depth 远近变化，验证底盘前进/后退距离保持。
4. 模拟 lost/无目标，验证目标丢失后底盘和云台安全停止。
5. 模拟 confidence 抖动，验证阈值、滤波和目标有效性判定。
6. 模拟目标慢速移动，验证整体输出平滑性和控制量不发散。

后续每次准备做视觉、硬件、控制器或安全层开发前，先提醒用户确认是否已经完成或需要补做上述 mock 场景验证。开发策略是先把控制方向、目标丢失停车、速度平滑和基础安全行为用 mock 数据确认，再接真实 YOLO、真实相机和真实底盘/云台。
